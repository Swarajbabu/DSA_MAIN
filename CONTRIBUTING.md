# 🤝 Contributing to DSA-PRACTICE

Thank you for your interest in contributing to the DSA-PRACTICE repository! This document provides guidelines and instructions for contributing.

## 📋 How to Contribute

### 1. Fork and Clone
```bash
# Fork the repository on GitHub
# Clone your fork
git clone https://github.com/YOUR-USERNAME/DSA-PRACTICE.git
cd DSA-PRACTICE
```

### 2. Create a Feature Branch
```bash
git checkout -b feature/your-feature-name
# or
git checkout -b bugfix/your-bug-fix
```

### 3. Make Your Changes

**For new solutions:**
- Place them in the appropriate category folder
- Include detailed comments explaining the approach
- Add time and space complexity analysis

**For bug fixes:**
- Fix the issue
- Test thoroughly
- Add a comment explaining the fix

**Code Style:**
- Use descriptive variable names
- Add comments for complex logic
- Follow standard C++ conventions
- Keep functions small and focused

### 4. Commit Your Changes
```bash
git add .
git commit -m "Add: Brief description of changes"
# Examples:
# "Add: Two pointer solution for array problem"
# "Fix: Memory leak in linked list"
# "Docs: Update README with new topic"
```

### 5. Push to GitHub
```bash
git push origin feature/your-feature-name
```

### 6. Create a Pull Request
- Go to the repository on GitHub
- Click "Compare & pull request"
- Provide a clear description of changes
- Reference any related issues

## ✅ PR Guidelines

- **Clear title**: Describe what the PR adds or fixes
- **Detailed description**: Explain the implementation or fix
- **Self-review**: Check your code before requesting review
- **Test**: Ensure all code compiles and runs correctly

## 📝 Commit Message Format

```
[Type]: Brief description

Type can be:
- Add: New feature, solution, or documentation
- Fix: Bug fix
- Improve: Performance improvement or code cleanup
- Docs: Documentation update
```

## 🎯 What We're Looking For

- ✅ Correct and optimized solutions
- ✅ Clear, readable code with comments
- ✅ Proper time/space complexity analysis
- ✅ New DSA topics or algorithms
- ✅ Documentation improvements
- ✅ Bug fixes
- ✅ Performance optimizations

## ❌ What to Avoid

- ✗ Incomplete or non-working code
- ✗ Poor code quality or unreadable logic
- ✗ Duplicate solutions without improvements
- ✗ Large files or unreasonable changes
- ✗ Merging before proper testing
